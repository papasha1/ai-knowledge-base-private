## GPIO 速度 vs 传感器时序

**记忆点：** MCU 太快，传感器跟不上，加延时就行。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** 用 STM32（72MHz）读 HX711 传感器，数据跳变严重，加了 `delayMicroseconds(1)` 后解决。

**原理：** STM32 的一次 GPIO 翻转只需约 14 纳秒，但 HX711 要求时钟脉冲至少 100 纳秒。MCU 太快了，翻完 GPIO 的时候传感器还没反应过来，数据读错了。

**规律：** 高速 MCU 对接慢速传感器时，几乎 always 需要在时序操作之间加延时。看数据手册的时序图，找最小脉宽参数（t_HIGH、t_LOW），对比 MCU 的 GPIO 翻转时间。

## 传感器预热与采样平均

**记忆点：** 上电等几秒再校准，多读几次取平均就稳了。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** HX711 上电后读数漂移，加了 2 秒预热 + 30 次采样去皮后稳定。

**原理：**
- **预热：** 模拟传感器刚上电时内部电路未稳定，输出会漂移。等几秒再校准是标准做法。
- **均值滤波：** 单次采样有随机噪声，多次采样取平均后噪声互相抵消。最简单的数字滤波方法。
- **中值滤波（进阶）：** 采 N 次排序取中间值，能排除偶发异常读数。

## 传感器校准的两种误差

**记忆点：** 偏移减固定值，比例除系数，两个都有就解方程。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** HX711 需要乘校准系数（比例误差），VL53L0X 需要减固定偏移（偏移误差）。

**原理：**
- **偏移误差：** 测量值 = 真实值 + 固定偏移。修正：减去偏移量。
- **比例误差：** 测量值 = 真实值 × 系数。修正：除以系数。
- **两点校准：** 两个测量点误差不同时，说明同时有两种误差，需解二元方程。

## STM32 printf 不支持 %f

**记忆点：** 浮点打印不出来，拆成整数拼小数点。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** 用 `hal_uart_printf("W:%.1fg", weight)` 输出重量，串口只显示 `W:g`，数字消失了。

**原理：** STM32 的 newlib 库默认不链接浮点 printf 支持（省 flash）。`%f` 和 `%.1f` 格式化会失败，返回空字符串。整数 `%d` 没问题。

**解决：** 把浮点拆成整数部分和小数部分，分别用 `%d` 打印。`int w_i = (int)weight; int w_d = (int)((weight - w_i) * 10) % 10;` 然后 `printf("%d.%d", w_i, w_d);`

**关联：** [GPIO 速度 vs 传感器时序](basics.md#gpio-速度-vs-传感器时序)（同样是对底层细节不了解踩的坑）

**来源：** FeedBowlForA，2026-05-21

**记忆点：** 卡住不动时电流最大，供电不够就烧或重启。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** MG995 舵机堵转电流 2.5A，STM32 的 5V 引脚只能供几百 mA，必须外部供电。

**原理：** 电机转动时产生反电动势，会限制电流。堵住不动时反电动势消失，电流飙升到最大值（堵转电流）。这个电流比正常运行大很多倍，如果电源扛不住，要么烧保险丝，要么电压跌落导致 MCU 重启。

**规律：** 选电源时要看堵转电流，不是空载电流。舵机、直流电机、步进电机都要注意。

**关联：** [PWM 与舵机控制](protocols.md#pwm-与舵机控制)

## PWM 输出必须配 GPIO 复用

**记忆点：** 定时器配好了但没输出，多半是 GPIO 没设成复用模式。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** STM32 TIM3_CH3 (PB0) 配了 PWM，舵机不转。原因：PB0 没有配置为 GPIO_MODE_AF_PP（复用推挽输出），PWM 信号出不来。

**原理：** STM32 的 GPIO 有多种模式：输入、输出、复用输出、模拟。PWM 是定时器的功能，不是 GPIO 本身的功能。要把 GPIO 引脚"复用"给定时器，必须把它设为 AF_PP（Alternate Function Push-Pull），否则定时器的信号走不到引脚上。

**做法：** 实现 `HAL_TIM_PWM_MspInit()` 回调函数，在里面配 GPIO。HAL 库在 `HAL_TIM_PWM_Init()` 时会自动调用这个回调。

**关联：** [PWM 与舵机控制](protocols.md#pwm-与舵机控制)

**来源：** FeedBowlForA，2026-05-21

**记忆点：** ST-Link 的 3.3V 不是供电，是"感知"目标在不在。

**掌握程度：** 🟡 理解原理，实操过一次

**场景：** STM32 Blue Pill 通过 5V 引脚供电，ST-Link 只接 SWDIO + SWCLK + GND，上传报 "Unable to set adapter speed"。把 ST-Link 的 3.3V 也接上后正常。

**原理：** ST-Link 有一个 VTref（目标参考电压）引脚，用来检测目标板的电压电平，以此配置 SWD 通信的信号幅度。如果 VTref 悬空，ST-Link 认为目标板没上电，拒绝通信。即使目标板已经通过其他方式供电了，VTref 也必须接到目标板的 3.3V 上。

**规律：** ST-Link 四根线都要接：SWDIO、SWCLK、GND、3.3V。3.3V 不是为了供电，是为了让 ST-Link 检测到目标。

## 主循环永远不能阻塞

**记忆点：** 嵌入式主循环一阻塞，按钮失灵、舵机卡死、屏幕冻结，全线崩盘。

**掌握程度：** 🟡 理解原理，踩过坑

**场景：** FeedBowlForA 主循环中 `hx711_read_weight_avg(20)` 阻塞 2 秒，导致按钮 2 秒才检测一次、舵机状态机 2 秒才更新、OLED 2 秒才刷新。按下按钮后系统完全没有反应。

**原理：** 嵌入式系统通常只有一个主循环（裸机）或少数几个任务（RTOS）。主循环负责所有模块的轮询：传感器、执行器、显示、通信。如果任何一个操作阻塞，其他所有模块都被卡住。阻塞时间越长，系统响应越差。

**正确做法：**
- 每次循环只做一小步，绝不等待
- 传感器用"轮询+就绪检查"：检查数据是否就绪，就绪就读，没就绪就跳过
- 执行器用"状态机+时间戳"：记录动作开始时间，每次循环检查是否完成
- 滤波用"后台累积"：每次循环采样 1 次存入缓冲区，读取时直接取平均值

## 用时间戳代替 delay

**记忆点：** delay 是"死等"，millis() 是"看表"——系统停不停摆就差在这。

**掌握程度：** 🟡 理解原理，踩过坑

**场景：** `feeder.cpp` 中关门用 `delay(500)`，这 500ms 内整个系统停摆，OLED 不刷新、按钮不响应。

**原理：** `delay()` 是忙等待（或 SysTick 等待），CPU 在这段时间什么都不做。而 `millis()` 返回系统运行时间，可以用它做"软定时器"：记录事件开始时间，每次循环检查是否到了目标时间。

**代码对比：**
```c
// 错误：阻塞 500ms
servo_close();
delay(500);
servo_stop();

// 正确：时间戳驱动
case CLOSING:
    if (first_entry) {
        servo_close();
        close_start_ms = millis();
    }
    if (millis() - close_start_ms >= 500) {
        servo_stop();
        state = IDLE;
    }
    break;
```

**规律：** 在主循环中，`delay()` 只能用于初始化阶段（上电等待传感器就绪）。运行阶段的所有定时都应该用 `millis()` 时间戳。

**关联：** [主循环永远不能阻塞](#主循环永远不能阻塞)（同一个问题的两个面）

**来源：** FeedBowlForA，2026-05-21

## 传感器采样要与业务逻辑解耦

**记忆点：** 传感器自己按节奏采样，业务逻辑按节奏读值，谁也不等谁。

**掌握程度：** 🟡 理解原理，踩过坑

**场景：** FeedBowlForA 中 HX711 每次采样 20 次（阻塞 2 秒），而业务逻辑需要每 50ms 更新一次。两者节奏不同步，硬塞在一起就出问题。

**原理：** 传感器的采样速率由硬件决定（HX711 10Hz、VL53L0X 可配置），业务逻辑的更新速率由需求决定（按钮防抖 50ms、状态机 100ms、显示 250ms）。两者节奏不同，不应该耦合。

**正确架构：**
```
传感器层（独立定时）：
  hx711_poll()  → 每 100ms 采样 1 次，存入环形缓冲区
  vl53l0x_poll() → 连续模式，后台自动测量

业务层（独立定时）：
  feeder_update() → 每 50ms 检查一次状态
  display_update() → 每 250ms 刷新一次

共享数据：
  hx711_get_weight()   → 直接返回滤波后的值（零阻塞）
  vl53l0x_get_distance() → 直接返回最新值（零阻塞）
```

**规律：** 如果传感器读取时间 > 业务逻辑更新周期，就必须解耦。用"后台采样 + 前台读值"的模式。

**关联：** [主循环永远不能阻塞](#主循环永远不能阻塞)、[传感器预热与采样平均](#传感器预热与采样平均)

## 防抖要考虑采样频率

**记忆点：** 防抖窗口比采样间隔还短，那防抖就是摆设。

**掌握程度：** 🟡 理解原理，踩过坑

**场景：** FeedBowlForA 中按钮防抖设为 50ms，但主循环因 HX711 阻塞每 2 秒才运行一次。用户按一下按钮（100-300ms），可能正好落在两次检测之间，完全漏掉。

**原理：** 防抖的原理是"两次检测之间如果状态一致，才认为有效"。如果采样间隔（2 秒）远大于防抖窗口（50ms），防抖逻辑根本触发不了——因为两次采样之间状态早就变了，防抖窗口毫无意义。

**正确做法：** 防抖时间 < 采样间隔 × 0.5。如果采样间隔无法缩短（比如传感器阻塞），应该用中断检测按钮，而不是在主循环中轮询。

**规律：** 按钮、编码器等用户输入设备，优先用外部中断 + 定时器防抖，不要依赖主循环轮询。

**关联：** [主循环永远不能阻塞](#主循环永远不能阻塞)（阻塞导致采样间隔变长）

## 先定义产品需求再选传感器

**记忆点：** 先想清楚"要解决什么问题"，再决定"用什么零件"，否则零件做出来也不知道怎么组合。

**掌握程度：** 🔴 只是知道概念

**场景：** FeedBowlForA 先做了 HX711 称重、VL53L0X 测距、OLED 显示，每个模块都能独立工作，但组合起来时发现：重量只显示不参与控制、距离只在 20 秒后检测一次、添粮模式不显示重量。传感器之间没有协作关系。

**原理：** 嵌入式项目的正确流程：
1. **定义产品**：这个设备要解决什么问题？用户怎么用？
2. **定义功能**：需要哪些功能？每个功能的输入/输出是什么？
3. **选择传感器**：每个功能需要什么传感器？传感器怎么协作？
4. **设计架构**：主循环怎么组织？状态机怎么设计？
5. **实现代码**：写驱动、写逻辑、调试

如果从第 4 步开始（先写驱动），到第 2 步时会发现传感器要么多了、要么少了、要么组合方式不对。

**规律：** "先做能用的零件，再想怎么组装"是嵌入式开发最常见的弯路。花 1 小时画架构图，能省 10 小时返工时间。
